Linux Fundamentals · 00
Shell — Origin & Types
Before writing scripts, understand WHAT a shell is, WHO created it, and WHY it matters for DevOps.
🐚
What is a Shell?
Interface · Interpreter · Bridge between User and Kernel
▼

💡 Intuition

Think of Linux like a car engine (the Kernel). You don't touch the engine directly — you use the steering wheel, pedals, and gear shift. The Shell is that interface — it takes your commands and passes them to the OS kernel, which then communicates with hardware.

⚙️
The Layers — Top to Bottom
You → type commands → Shell (interprets) → Kernel (executes) → Hardware (CPU, RAM, Disk)

⚙️ Technical Definition

A Shell is a Command Interpreter — it reads the commands you type, interprets them, and asks the kernel to execute them. It acts as the interface between the user and the OS (Linux Kernel).

ShellCreatorYearKey Use
sh (Bourne Shell)Stephen Bourne1979The original Unix shell. Pioneered scripting. Location: /bin/sh
bash (Bourne Again SH)Brian Fox (GNU)1989Most common today. Default in Ubuntu/Debian. Modern DevOps scripting tool.
zshPaul Falstad1990Default on macOS. More features than bash.
sh (dash)——Lightweight POSIX sh, faster for system scripts

🔑 Why This Matters for DevOps

1
Bourne Shell (sh) started shell scripting. It proved you could automate tasks by writing commands in a file and running them.
2
Bash extended sh with modern features — arrays, better string handling, process substitution. It's the default choice for all DevOps scripts.
3
Shell scripting = automation ka base — CI/CD pipelines, deployment scripts, health checks, backups — all of it runs in shell.
💀
Pro Tip — sh vs bash
/bin/sh on Ubuntu is actually dash (not bash) — it's faster but missing bash features. Always write #!/bin/bash at the top of your scripts to guarantee bash is used. Never assume sh = bash.
📄
Writing Your First Script — The Essentials
#!/bin/bash · .sh · chmod +x · ./script.sh
▼

💡 Intuition

A shell script is just a text file with a list of commands. But for it to be a proper, runnable script, 4 things need to happen:

1
Shebang line — #!/bin/bash must be the very first line. It tells the OS which interpreter to use. Without it, behavior is unpredictable.
2
.sh extension — Save the file as script.sh. The extension is a convention (Linux doesn't require it) but makes it clear this is a shell script.
3
Make it executable — Run chmod +x script.sh. Without this, Linux refuses to run it as a program.
4
Run it — Use ./script.sh (the ./ means "in current directory"). Or use the full path.
bash — complete first script workflow
# Step 1: Create the file
nano myscript.sh

# Step 2: Write inside the file
#!/bin/bash          # ← mandatory shebang. BEST PRACTICE.
echo "Hello DevOps!"
echo "Today is: $(date)"

# Step 3: Make executable
chmod +x myscript.sh

# Step 4: Run it
./myscript.sh

# Alternative: run with bash directly (no chmod needed)
bash myscript.sh
⚠️
Why ./ before the script?
Linux only looks for programs in directories listed in $PATH. Your current folder is NOT in PATH by default. The ./ prefix tells bash: "look in THIS folder, not PATH". Without it, bash says "command not found".

Basic Commands
The commands you'll type a thousand times as a DevOps engineer. These aren't just syntax — they're how you think on a server.
📁
Navigation & File Management
ls · cd · pwd · mkdir · rm · cp · mv · find
▼

💡 Intuition

Think of the filesystem as a tree. pwd tells you where you are. ls shows what's around you. cd moves you. Every DevOps task — deploying code, checking logs — starts with navigating to the right place.

⚙️ Technical

CommandSyntaxWhat it does
lsls -la /pathList files (-l = long format, -a = hidden files)
cdcd /var/logChange directory. cd ~ = home, cd - = previous dir
pwdpwdPrint working directory (where are you)
mkdirmkdir -p a/b/cMake directory. -p creates parents if missing
rmrm -rf dir/Remove. -r = recursive, -f = force (no prompt)
cpcp -r src/ dst/Copy files or directories
mvmv old newMove or rename
findfind / -name "*.log"Search for files by name, type, size, time

💻 Examples

bash
# List all files including hidden, human-readable sizes
ls -lah /var/log/

# Find all .log files modified in last 7 days
find /var/log -name "*.log" -mtime -7

# Find files larger than 100MB
find / -size +100M -type f 2>/dev/null

# Create nested dirs in one shot
mkdir -p /opt/myapp/{logs,config,scripts}

# Safe remove: preview before deleting
find . -name "*.tmp" -exec echo {} \;
# Then actually delete:
find . -name "*.tmp" -delete
❌
Common Mistake
rm -rf / or rm -rf /* — these delete the ENTIRE system. Always double-check your path before running rm -rf. Real engineers have lost production servers this way.
💀
Pro Tips
Use ls -lah instead of just ls — always. The 'h' flag shows human-readable sizes. Use cd - to toggle between two directories instantly. Use find ... -exec for batch operations on files.
🔍
Text Searching & Manipulation
grep · awk · sed · cut · sort · uniq · wc
▼

💡 Intuition

In DevOps, you'll constantly search through logs, config files, and outputs. grep is your most-used search tool. awk and sed are for transforming text. Think of them as spreadsheet operations, but in the terminal.

💻 Examples

bash
# grep: search for pattern in file
grep "ERROR" /var/log/app.log
grep -i "error" app.log        # case insensitive
grep -r "DB_HOST" /etc/          # recursive search
grep -v "DEBUG" app.log         # invert: lines NOT matching
grep -n "ERROR" app.log         # show line numbers
grep -c "ERROR" app.log         # count matching lines

# awk: column-based processing
awk '{print $1, $3}' file.txt   # print col 1 and 3
awk -F: '{print $1}' /etc/passwd # use : as delimiter → usernames

# sed: stream editor (find & replace)
sed 's/old/new/g' file.txt       # replace in output
sed -i 's/localhost/prod-db/g' config.env  # replace IN FILE

# Practical: count unique IPs in nginx log
awk '{print $1}' /var/log/nginx/access.log | sort | uniq -c | sort -rn | head 10
⚠️
Watch Out
sed -i modifies files in-place permanently. Always test without -i first, or use sed -i.bak to create a backup.
💀
Pro Tips
Chain grep: grep "ERROR" app.log | grep -v "timeout" — find errors but exclude timeouts. Use grep -A 5 "ERROR" to see 5 lines AFTER the match (context!).
👁️
Viewing Files & Logs
cat · less · head · tail · tail -f
▼

💡 Intuition

In production, you need to watch logs in real time. tail -f is the single most important command for debugging live systems — it "follows" a log as it grows.

bash
cat file.txt              # dump entire file
less largefile.log        # paginate (q to quit, / to search)
head -20 file.txt         # first 20 lines
tail -50 file.txt         # last 50 lines
tail -f /var/log/app.log  # LIVE follow — use this for debugging!
tail -f app.log | grep "ERROR"  # live-follow + filter
💀
Pro Tip
Never use cat on large files — it'll flood your terminal. Use less instead. For log monitoring: tail -f app.log | grep --line-buffered "ERROR".
Linux Fundamentals · 02
File Permissions
Understanding who can read, write, and execute files. This is critical for security and deployments.
🔐
chmod — Symbolic & Numeric
rwx · 755 · 644 · owner/group/others
▼

💡 Intuition

Every file has 3 audiences: the owner, the group, and others (everyone else). Each audience gets 3 permissions: read, write, execute. Permissions protect your scripts and config files from unauthorized access.

⚙️ The Permission Grid

permission structure
  -  rwx  r-x  r--
  |   |    |    |
  |   |    |    └── Others permissions
  |   |    └─────── Group permissions
  |   └──────────── Owner permissions
  └──────────────── File type (- = file, d = dir, l = symlink)

  r = 4, w = 2, x = 1   → Add them up for numeric mode
  rwx = 7,  r-x = 5,  r-- = 4,  rw- = 6,  --- = 0

💻 chmod Examples

bash
# Numeric (octal) mode
chmod 755 script.sh    # owner=rwx, group=rx, others=rx → typical script
chmod 644 config.txt   # owner=rw, group=r, others=r → typical config
chmod 600 private.key  # owner=rw, nobody else → SSH keys MUST be this
chmod 700 secret.sh    # owner=rwx, nobody else

# Symbolic mode (more readable)
chmod +x script.sh      # add execute for all
chmod u+x script.sh     # add execute for owner only
chmod o-w file.txt      # remove write from others
chmod go-rwx secret.key # remove all perms from group and others
chmod -R 755 /opt/app/  # recursive: apply to all files in dir

# chown: change ownership
chown ubuntu:ubuntu file.txt      # user:group
chown -R www-data:www-data /var/www/ # typical nginx setup

🔢 Quick Reference: Common Permission Numbers

NumberSymbolicUse Case
777rwxrwxrwx⚠️ NEVER use in production — anyone can do anything
755rwxr-xr-xShell scripts, executables, directories
644rw-r--r--Config files, HTML, text files
600rw-------SSH private keys, secrets
400r--------Read-only keys (AWS key files)
700rwx------Private scripts
❌
Common Mistake — chmod 777
Never set 777 in production. This means ANYONE on the system (and sometimes anyone on the internet) can read, write, and execute your files. SSH keys with wrong permissions will be outright rejected by OpenSSH.
💀
Pro Tip
Use stat filename to see permissions in both symbolic and octal form. Use ls -la to see symbolic permissions for all files in a directory.
Linux Fundamentals · 03
Processes
Managing running programs — how to monitor, control, and kill processes on a server.
⚙️
ps, top, kill & Background Jobs
ps aux · top · htop · kill · & · jobs · nohup
▼

💡 Intuition

Every program running on Linux is a process with a unique PID (Process ID). In DevOps, you need to find processes eating CPU/memory, kill stuck processes, and run services in the background without them dying when you log out.

bash
# ps: snapshot of running processes
ps aux                    # all processes, all users, full detail
ps aux | grep nginx       # find nginx processes
ps -ef | grep python      # another format

# top: live, interactive process viewer
top                       # press q to quit, P to sort by CPU, M by memory
htop                      # prettier version (install separately)

# kill: send signal to process
kill 1234                  # SIGTERM (15) → polite shutdown, process can clean up
kill -9 1234              # SIGKILL → force kill, NO cleanup possible
killall nginx              # kill all processes named nginx
pkill -f "python app.py"  # kill by matching command string

# Background jobs
sleep 60 &                 # run in background, get PID
jobs                       # list background jobs in this shell
fg %1                      # bring job 1 to foreground
bg %1                      # resume job 1 in background
Ctrl+Z                     # suspend foreground process
Ctrl+C                     # terminate foreground process

# nohup: keep running after logout
nohup python3 server.py &       # runs even after SSH disconnect
nohup ./deploy.sh & > deploy.log 2>&1  # capture all output
⚠️
kill -9 vs kill (Signal 15)
Always try kill PID first. Only use kill -9 if the process won't stop. kill -9 doesn't allow the process to save data or clean up — databases can corrupt.
💀
Pro Tips
To find PID of a process using a port: lsof -i :8080 or ss -tlnp | grep 8080. Use pgrep nginx to just get PIDs.
Linux Fundamentals · 04
Redirection & Pipes
The superpower of Unix — chaining commands together and controlling where output goes.
🔀
stdin, stdout, stderr & Pipes
> · >> · | · 2>&1 · /dev/null · tee
▼

💡 Intuition

Every command has 3 streams: stdin (input, fd=0), stdout (output, fd=1), stderr (errors, fd=2). You can redirect any stream to a file or to another command. The pipe (|) connects stdout of one command to stdin of the next — this is how you build powerful one-liners.

bash
# OUTPUT REDIRECTION
echo "hello" > file.txt     # write (overwrites!)
echo "hello" >> file.txt    # append (safe)

# ERROR REDIRECTION
ls /nonexistent 2> err.txt   # redirect stderr to file
ls /nonexistent 2>&1         # merge stderr into stdout

# CAPTURE BOTH stdout and stderr
./deploy.sh > output.log 2>&1   # everything to one file

# SUPPRESS errors
find / -name "secret" 2>/dev/null  # silence permission denied

# PIPES
ps aux | grep python | awk '{print $2}'   # PIDs of python processes
cat /etc/passwd | cut -d: -f1 | sort       # sorted usernames

# tee: write to file AND stdout simultaneously
./build.sh | tee build.log      # see output AND save to file

# xargs: convert stdin to command arguments
cat servers.txt | xargs ping -c1  # ping each server in the file
find . -name "*.log" | xargs rm   # delete all found logs
❌
Common Mistake — Order Matters for 2>&1
./script.sh 2>&1 > out.log is WRONG. Stderr won't go to the file. Correct: ./script.sh > out.log 2>&1 — redirect stdout first, then merge stderr into it.

Compression & Archives
tar · gz · zip · unzip
▼
📖
tar vs gz — what's the difference?
tar = bundles multiple files into ONE file (like a folder in a box). It does NOT compress by default. gz (gzip) = compresses a file to make it smaller. tar.gz = bundled AND compressed. That's why it's the most common archive format on Linux.
bash
# tar: bundle files
tar -cvf archive.tar folder/       # c=create, v=verbose, f=filename
tar -czvf archive.tar.gz folder/   # z=compress with gzip
tar -xzvf archive.tar.gz          # x=extract
tar -tf archive.tar.gz             # list contents without extracting

# zip/unzip: simpler, cross-platform
zip output.zip file.txt           # compress single file
zip -r output.zip folder/         # compress a folder (-r = recursive)
unzip output.zip                  # decompress/extract
unzip -l output.zip               # list contents

Soft Links & Hard Links
Two ways to create "shortcuts" to files in Linux — understanding the difference is a classic interview question.
🔗
Soft Link vs Hard Link — Side by Side
ln -s · ln · inode · symlink
▼

💡 Intuition

Think of a file like a house. The actual data (house) lives at an inode (address on disk). A hard link is another door to the SAME house — the house exists as long as ANY door exists. A soft link is a signpost pointing to the house's address — if the house is demolished, the signpost still exists but leads nowhere.

FeatureSoft Link (Symbolic)Hard Link
Commandln -s source linkln source link
Points toPath (file name)Inode (actual data)
Source deleted?Link BREAKS (dangling)Data still accessible
Cross filesystem?✅ Yes❌ No
Works on dirs?✅ Yes❌ No (mostly)
Shows aslrwxrwxrwx (l prefix in ls)Same as regular file
Use caseShortcuts, versioned deploysBackups, inode-level duplicates
bash
# SOFT LINK (Symbolic Link)
# Source file affected = soft link breaks if source is deleted
ln -s /opt/app-v2.3/ /opt/current   # create symlink
ls -la /opt/current                   # shows: current -> /opt/app-v2.3/

# Real DevOps use: switch versions with one command
ln -sfn /opt/app-v2.4/ /opt/current  # atomic version switch!

# HARD LINK
# Duplicate file that isn't affected by source file deletion
ln /original/file.txt /backup/file.txt
# Now both point to the same data (same inode)
# Deleting /original/file.txt = data still in /backup/file.txt

# Check inode numbers (same inode = hard link)
ls -i /original/file.txt /backup/file.txt
💀
Pro Tip — Real DevOps Use of Soft Links
The zero-downtime deploy technique uses symlinks: keep /opt/current as a symlink to the active version. To deploy, extract new code to /opt/v2.4/, then run ln -sfn /opt/v2.4 /opt/current — the switch is atomic (instant). This is how many production deployments work.
Shell Scripting · 01
Variables & Input
The foundation of any shell script — storing data, reading user input, and using special variables.
📦
Variables, Special Variables & Input
$VAR · $# · $@ · $? · $0 · read · export
▼

💡 Intuition

Shell variables are like sticky notes. You write something down (NAME="Alice"), and later you can read it back (echo $NAME). No spaces around = — ever. That's the #1 beginner mistake.

bash
#!/bin/bash
# VARIABLES
NAME="Alice"                    # no spaces around =
AGE=25
echo "Hello, $NAME. You are $AGE."
echo "Hello, ${NAME}!"            # braces for clarity

# READ INPUT
read -p "Enter your name: " username
echo "Welcome, $username"
read -s -p "Password: " pass      # -s = silent (no echo)

# COMMAND OUTPUT as variable
TODAY=$(date +%Y-%m-%d)          # preferred syntax
FILES=$(ls /var/log/*.log | wc -l)
echo "Today: $TODAY, Logs: $FILES"

# SPECIAL VARIABLES
echo "Script name: $0"
echo "First arg: $1"
echo "All args: $@"
echo "Arg count: $#"
echo "Last exit code: $?"
echo "Current PID: $$"

# EXPORT: make available to child processes
export DB_HOST="prod-db.internal"

# DEFAULT VALUES
ENV=${1:-"production"}           # use $1 or "production" if not set
LOG=${LOG_FILE:-"/var/log/app.log"}
VariableMeaning
$0Script name
$1, $2...Positional arguments
$@All arguments as separate words
$#Number of arguments
$?Exit code of last command (0=success)
$$PID of current script
$!PID of last background process
❌
Common Mistakes
NAME = "Alice" → WRONG (spaces around =). NAME=Alice Smith → WRONG (no quotes around value with spaces). Always quote values that might have spaces: NAME="Alice Smith"
Shell Scripting · 02
Conditions — if & case
Make your scripts smart — branch based on values, file existence, or command success.
🌿
if / elif / else + case
[ ] · [[ ]] · -f · -d · -z · -eq · -gt
▼

💡 Intuition

In shell, if doesn't check "true/false" — it checks if a command succeeds (exit 0). The [ and [[ are actually commands that return 0 or 1. This trips up everyone at first.

bash
#!/bin/bash

# BASIC if
if [[ $1 == "prod" ]]; then
    echo "Deploying to production!"
elif [[ $1 == "staging" ]]; then
    echo "Deploying to staging"
else
    echo "Unknown environment"
    exit 1
fi

# FILE/DIR TESTS
if [[ -f "/etc/nginx/nginx.conf" ]]; then
    echo "Config exists"
fi
if [[ -d "/opt/app" ]]; then echo "Dir exists"; fi
if [[ ! -f ".env" ]]; then
    echo "ERROR: .env missing"; exit 1
fi

# NUMBER COMPARISONS
COUNT=5
if [[ $COUNT -gt 3 ]]; then echo "High count"; fi

# STRING CHECKS
if [[ -z "$VAR" ]]; then   # -z = empty string
    echo "VAR is empty!"; exit 1
fi
if [[ -n "$VAR" ]]; then   # -n = non-empty
    echo "VAR is set"
fi

# CASE statement (cleaner than many elif)
case "$ENV" in
    prod|production)
        SERVER="prod.example.com";;
    staging)
        SERVER="staging.example.com";;
    dev*)                        # wildcard match
        SERVER="localhost";;
    *)
        echo "Unknown env"; exit 1;;
esac
Test FlagMeaning
-f FILEFile exists and is a regular file
-d DIRDirectory exists
-z "$VAR"String is empty
-n "$VAR"String is NOT empty
-eq, -neNumber equal, not equal
-gt, -ltGreater than, less than
-ge, -leGreater/less than or equal

🔢 Number Operators vs String Operators — The Key Distinction

⚠️
Most Common Interview Trap in Scripting
Number comparison and string comparison use DIFFERENT operators. Using == for numbers or -eq for strings produces wrong results or errors. Know both sets cold.
🔢 Only for NUMBERS — Numeric Comparison
OperatorMeaning
-eqequal     → [[ $a -eq $b ]]
-nenot equal → [[ $a -ne $b ]]
-gtgreater than → [[ $a -gt $b ]]
-ltless than → [[ $a -lt $b ]]
-gegreater than or equal → [[ $a -ge $b ]]
-leless than or equal → [[ $a -le $b ]]
🔤 Only for STRINGS — String Comparison
OperatorMeaning
==equal (strings match) → [[ "$a" == "$b" ]]
!=not equal → [[ "$a" != "$b" ]]
<alphabetically before → [[ "$a" < "$b" ]]
>alphabetically after → [[ "$a" > "$b" ]]
bash — correct usage
# NUMBERS — use arithmetic flags (-eq, -gt, -lt...)
COUNT=10
if [[ $COUNT -gt 5 ]]; then echo "Big count"; fi
if [[ $COUNT -eq 10 ]]; then echo "Exactly 10"; fi

# STRINGS — use == and !=
ENV="prod"
if [[ "$ENV" == "prod" ]]; then echo "Production!"; fi
if [[ "$ENV" != "dev" ]]; then echo "Not dev"; fi

# ❌ WRONG — mixing them
if [[ $COUNT == 10 ]]; then   # string comparison on a number — works but fragile
if [[ "$ENV" -eq "prod" ]]; then   # ERROR: -eq needs integers
⚠️
[ ] vs [[ ]]
Use [[ ]] (double brackets) always in bash scripts. It's smarter — supports regex, doesn't break on empty variables, handles &&/|| inside. Single [ ] is POSIX sh and has many gotchas.

Shell Scripting · 05
Error Handling
Write scripts that fail safely and loudly — not silently and destructively.
🛡️
set -e, set -u, pipefail & exit codes
$? · || · && · trap · set -euo pipefail
▼

💡 Intuition

By default, bash scripts keep running even when a command fails. This is dangerous in deployment scripts. set -e makes your script stop at the first error — like a safety net.

bash
#!/bin/bash
# PRODUCTION-READY header — always use this combo
set -euo pipefail
# -e: exit on any error
# -u: error on unset variables
# -o pipefail: pipe fails if ANY command in it fails

# EXIT CODES: 0 = success, anything else = failure
ls /tmp
echo "Exit code: $?"   # 0
ls /nonexistent
echo "Exit code: $?"   # 2 (with set -e, script stops here!)

# OR operator: run second cmd if first fails
mkdir /opt/app || { echo "mkdir failed!"; exit 1; }

# AND operator: run second only if first succeeds
cd /opt/app && git pull origin main

# TRAP: run cleanup on exit or error
trap 'echo "Script failed at line $LINENO"; cleanup' ERR
trap 'rm -f /tmp/lockfile' EXIT    # always runs on exit

cleanup() {
    echo "Cleaning up..."
    rm -f /tmp/deploy.lock
}

# HANDLE specific command failures
if ! systemctl start myapp 2>/dev/null; then
    echo "ERROR: Failed to start service" >&2
    journalctl -u myapp -n 20 >&2
    exit 1
fi
💀
Pro Tip — The Golden Header
Every production script should start with exactly: #!/bin/bash then set -euo pipefail. This single line prevents 80% of script disasters.
⚠️
set -e gotcha
With set -e, a failing command in an if condition won't trigger exit (that's correct behavior — that's the test). But a standalone failing command will. Use command || true to intentionally ignore a failure.
Shell Scripting · 06
Quoting Rules
The most misunderstood part of shell scripting — and the source of countless bugs.
✏️
" vs ' vs no quotes
double · single · unquoted · $() · word splitting
▼
Quote TypeVariables?Use Case
"double"✅ ExpandedUse for most cases — variables work, spaces preserved
'single'❌ LiteralUse when you want NO substitution at all
no quotes✅ + splitAvoid — causes word splitting and glob expansion
bash
NAME="John Doe"
FILE="my file.txt"

# Double quotes: variables expand, spaces preserved
echo "Hello, $NAME"       # Hello, John Doe

# Single quotes: NOTHING expands — literal string
echo 'Hello, $NAME'       # Hello, $NAME

# No quotes: word splitting! This BREAKS with spaces in filename
cp $FILE /tmp/            # ERROR: treats "my" and "file.txt" as separate args
cp "$FILE" /tmp/          # CORRECT

# In sed commands with variable: must use double quotes
OLD="localhost"; NEW="prod-db"
sed "s/$OLD/$NEW/g" config.txt   # variables expand in sed pattern
sed 's/$OLD/$NEW/g' config.txt   # WRONG: literally "$OLD"

# Mixing: when you need both
echo "Price: \$5 for $NAME"  # escape $ to print literal, or use variable
💀
Golden Rule
Always double-quote your variables. "$variable" — never just $variable. This alone will save you from 90% of shell scripting bugs involving spaces or special characters.
Git & GitHub · 01
Git — Core Concepts

📖
What is Git? What is GitHub?
Version Control · Local vs Remote · Linus Torvalds 2005
▼

💡 Intuition

Imagine writing a novel and saving a copy every time you finish a chapter — so you can go back if you mess something up. Git does exactly this for code. It tracks every change, who made it, and when. You can rewind time.

🏛️
Who Created What
Git — Created by Linus Torvalds in 2005 (same person who created the Linux kernel). He built it in 2 weeks to manage Linux kernel development.

GitHub — Created by Tom Preston-Werner, Chris Wanstrath, and PJ Hyett in San Francisco. It's a website that hosts Git repositories online.

⚙️ Git vs GitHub — Not the Same Thing

Analogy: Git is like Microsoft Word's "Track Changes" feature. GitHub is like Google Drive — where you store and share the document.

🗺️
The 4 Areas of Git
Working Directory · Staging Area · Local Repo · Remote Repo
▼

💡 Intuition

Git has 4 distinct "zones" that your code moves through. Understanding this is the key to understanding every git command.

1
💻 Working Directory — The project folder on your computer. When you create, edit, or delete files, changes happen here first. Git tracks them but hasn't saved them yet.
2
📋 Staging Area — Where you prep changes before committing. Like "I'm almost ready! Let me pick which changes to include." Use git add to move changes here. You choose WHAT to commit.
3
🏡 Local Repository — A Git repo that lives on your own computer. Use git commit to permanently save staged changes here with a message. This is your local history.
4
☁️ Remote Repository — A GitHub repo stored on the internet (for the latest version). Use git push to upload your local commits here. This is the source of truth for teams.
the flow
# The journey of a file change:

Working Directory  →(git add)→  Staging Area  →(git commit)→  Local Repo  →(git push)→  Remote (GitHub)

# Reverse direction:

Remote (GitHub)  →(git clone/pull)→  Local Repo  →(git checkout)→  Working Directory
🍴
Forking
Forking is creating a personal copy of someone else's repository on GitHub. You can modify your fork freely without affecting the original. Used in open-source contributions — fork → make changes → send a Pull Request back.

Culture · Automation · Measurement · Sharing
▼

CI/CD Pipeline Flow

▼

A CI/CD pipeline is like an assembly line for code. Each station (stage) does one job. If any station fails, the line stops and alerts you. Code only reaches production if it passes every station.

GitHub Actions — .github/workflows/deploy.yml
name: Deploy Application

on:
  push:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Install dependencies
        run: npm install
      
      - name: Run tests
        run: npm test
      
      - name: Build Docker image
        run: docker build -t myapp:${{ github.sha }} .
      
      - name: Push to registry
        run: docker push myapp:${{ github.sha }}

  deploy:
    needs: build-and-test   # only runs if previous job passes
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to server
        run: |
          ssh deploy@${{ secrets.SERVER_IP }} \
            "docker pull myapp:${{ github.sha }} && \
             docker stop myapp && \
             docker run -d --name myapp myapp:${{ github.sha }}"

Health Check Script
Checks services + disk + memory, sends alerts
▼
bash — health_check.sh
#!/bin/bash
set -euo pipefail

SERVICES=("nginx" "postgresql" "redis")
DISK_THRESHOLD=80
MEM_THRESHOLD=90
ALERT_EMAIL="ops@company.com"
FAILED=0

log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*"; }

# Check services
for service in "${SERVICES[@]}"; do
    if systemctl is-active --quiet "$service"; then
        log "✓ $service is running"
    else
        log "✗ $service is DOWN — attempting restart"
        systemctl restart "$service" || FAILED=$((FAILED+1))
    fi
done

# Check disk usage
DISK_USAGE=$(df / | awk 'NR==2 {print $5}' | tr -d '%')
if [[ $DISK_USAGE -gt $DISK_THRESHOLD ]]; then
    log "⚠ Disk usage: ${DISK_USAGE}% (threshold: ${DISK_THRESHOLD}%)"
    FAILED=$((FAILED+1))
fi

# Check memory
MEM_USAGE=$(free | awk '/Mem:/ {printf "%.0f", $3/$2*100}')
if [[ $MEM_USAGE -gt $MEM_THRESHOLD ]]; then
    log "⚠ Memory usage: ${MEM_USAGE}%"
    FAILED=$((FAILED+1))
fi

if [[ $FAILED -gt 0 ]]; then
    log "ALERT: $FAILED issue(s) found"
    mail -s "Server Alert: $FAILED issues" "$ALERT_EMAIL" <<< "Check logs!"
    exit 1
fi

log "All checks passed ✓"
🚀
Zero-Downtime Deployment Script
Blue-green style deploy with rollback
▼
bash — deploy.sh
#!/bin/bash
set -euo pipefail

APP_NAME="myapp"
NEW_VERSION="${1:?ERROR: Version required}"
DEPLOY_DIR="/opt/deploys"
CURRENT_LINK="/opt/current"
NEW_DIR="${DEPLOY_DIR}/${NEW_VERSION}"

log() { echo "[$(date '+%H:%M:%S')] [DEPLOY] $*"; }

rollback() {
    log "ROLLBACK triggered — reverting..."
    if [[ -L "${CURRENT_LINK}.prev" ]]; then
        ln -sfn "$(readlink ${CURRENT_LINK}.prev)" "$CURRENT_LINK"
        systemctl restart "$APP_NAME"
        log "Rolled back successfully"
    fi
}

trap rollback ERR

log "Starting deploy of version: $NEW_VERSION"

# 1. Create new version directory
mkdir -p "$NEW_DIR"
cp -r /tmp/release/* "$NEW_DIR/"

# 2. Save current link as previous
[[ -L "$CURRENT_LINK" ]] && cp -P "$CURRENT_LINK" "${CURRENT_LINK}.prev"

# 3. Switch symlink atomically
ln -sfn "$NEW_DIR" "$CURRENT_LINK"

# 4. Restart service
systemctl restart "$APP_NAME"

# 5. Health check (with retry)
for i in {1..12}; do
    if curl -sf http://localhost:8080/health; then
        log "Health check passed ✓"
        break
    fi
    [[ $i -eq 12 ]] && { log "Health check FAILED after 60s"; exit 1; }
    log "Waiting for service... ($i/12)"
    sleep 5
done

log "Deploy complete! Running version: $NEW_VERSION"

Avoid These
Beginner Mistakes
The actual mistakes that have caused production outages. Learn them before you make them.
❌
rm -rf with wrong path
Running rm -rf $DIR/ where $DIR is empty deletes your current directory. Always validate variables before using in destructive commands: [[ -n "$DIR" ]] || exit 1
❌
Not quoting variables
cp $FILE /backup/ breaks when $FILE has spaces. Always use cp "$FILE" /backup/. This is responsible for countless subtle bugs.
❌
No set -euo pipefail
Scripts without error handling continue running after failures, causing cascading disasters. Your deploy script might "succeed" even though it failed halfway through.
❌
chmod 777 everything
"It works now!" — yes, and you've just opened a security hole. Use 755 for scripts, 644 for files, 600 for secrets. Think about WHY each file needs what permissions.
❌
Hardcoding credentials in scripts
PASSWORD="secret123" in a script pushed to git. Use environment variables or secret management tools. Check git history — deleted secrets can still be found.
❌
kill -9 as first resort
SIGKILL doesn't allow cleanup. Databases can corrupt, files can get stuck mid-write. Always try kill PID (SIGTERM) first and wait a few seconds.
❌
sed -i without testing first
Running sed -i 's/old/new/g' /etc/nginx/nginx.conf without first running sed 's/old/new/g' /etc/nginx/nginx.conf to preview. Always preview first!
❌
Using cat to view huge log files
cat /var/log/huge-app.log on a 2GB file will flood your terminal and may crash your session. Use less or tail -n 100.
Interview Intelligence
Common Interview Traps
Questions designed to catch people who memorized without understanding.
🪤
"What's the difference between > and >> ?"
Expected answer: > overwrites the file (truncates first). >> appends. Trap: people say "redirect vs append" — you need to say OVERWRITE vs append. Follow-up: what happens to the file if you do echo "" > file? (It creates an empty file / wipes it.)
🪤
"What does 2>&1 mean and does order matter?"
Trap: many say "yes order matters" but can't explain why. Answer: redirect stdout first, THEN redirect stderr to wherever stdout now points. cmd > out.log 2>&1 is correct. cmd 2>&1 > out.log sends stderr to original stdout (terminal), not the file.
🪤
"What is $@ vs $* in shell scripts?"
Both expand to all arguments. Trap is in quoting: "$@" preserves each argument separately (correct). "$*" merges all args into one string. Use "$@" always when passing args to another command.
🪤
"chmod 644 vs 755 — when do you use which?"
Trap: people guess. Rule: does it need to be EXECUTED? Use 755. Is it just data? Use 644. Secret/key? 600. Directories need execute (x) permission to be entered, so dirs should be 755 or 700, never 644.
🪤
"What happens when you run a command that fails in a pipeline?"
Trap: without set -o pipefail, the pipeline's exit code is the LAST command's exit code. So failing_cmd | echo "done" exits 0 even though it failed! With pipefail, any failure in the pipe counts.
🪤
"What's the difference between CI and CD?"
CI = Continuous Integration (merge + build + test automatically). CD can mean Continuous Delivery (auto-deploy to staging, human approves prod) OR Continuous Deployment (auto-deploys all the way to prod). Clarify which CD when asked.
🪤
"How do you check if a service is running in a script?"
Trap: people say ps aux | grep nginx. Problem: this always finds itself (grep process). Correct answer: systemctl is-active --quiet nginx — clean, no false positives, proper exit code. Or use pgrep -x nginx.

🐧 Linux Questions

Q
What does the command chmod 755 script.sh do? Explain each digit.
▼

chmod 755 sets permissions using octal notation. The digits represent owner, group, and others respectively. 7 = rwx (4+2+1), 5 = r-x (4+0+1), 5 = r-x. So the owner can read, write, and execute; group and others can only read and execute. This is the standard permission for shell scripts that others need to run but not modify.

Q
How do you find all files modified in the last 24 hours in /var/log?
▼

find /var/log -type f -mtime -1 — The -mtime -1 means "less than 1 day old" (modified in last 24 hours). -type f ensures only files, not directories. You can also use -mmin -60 for the last 60 minutes.

Q
What's the difference between a hard link and a soft (symbolic) link?
▼

A hard link points directly to the inode (the actual data on disk). Multiple hard links to the same file all see the same data; deleting one doesn't delete the data until all hard links are gone. They can't span filesystems. A soft/symlink is a pointer to a path — if the target is deleted, the symlink breaks. Symlinks can span filesystems and point to directories.

Q
How do you check which process is using port 8080?
▼

ss -tlnp | grep 8080 or lsof -i :8080. The ss approach is preferred on modern Linux. It shows the process name and PID directly. lsof is more portable. Either is acceptable in interviews.

📜 Shell Scripting Questions

Q
What does set -euo pipefail do and why should every script have it?
▼

-e: Exit immediately when any command fails (non-zero exit). -u: Treat unset variables as errors (prevents bugs from typos in variable names). -o pipefail: Pipe fails if ANY command in the pipeline fails, not just the last one. Together, they make scripts fail loudly and early rather than silently continuing after errors — critical for deployment and automation scripts.

Q
What's wrong with this code: for file in $(ls /tmp/*.log); do rm $file; done?
▼

Multiple problems: (1) $(ls ...) output is word-split, so filenames with spaces break. (2) $file is unquoted in rm, causing word splitting issues. (3) It's completely unnecessary — just use rm /tmp/*.log directly (glob expansion). Or if you need to process each file: for file in /tmp/*.log; do rm "$file"; done.

Q
How do you pass arguments to a shell script and validate them?
▼

Arguments are available as $1, $2, etc. $# is the count. $@ is all arguments. For validation: [[ $# -lt 1 ]] && { echo "Usage: $0 <env>"; exit 1; }. Or use the built-in error form: ENV="${1:?ERROR: environment argument required}" — this exits with the error message if $1 is empty or unset.

🚀 DevOps Conceptual Questions

Q
Explain CI/CD in simple terms. What's the practical benefit?
▼

CI/CD is an automated pipeline: developer pushes code → CI server automatically builds and tests it → if tests pass, CD automatically deploys it. Practical benefits: bugs caught early (in dev, not production), faster release cycles (deploy multiple times per day vs once per month), consistent deployments (no "works on my machine"), and easy rollbacks. The key insight: frequent small changes are less risky than rare large changes.

Q
What's the difference between a container and a virtual machine?
▼

A VM runs a full OS (kernel + userspace) on virtualized hardware — takes minutes to start, uses GBs of memory. A container shares the host OS kernel, only packaging the application and its dependencies — starts in seconds, uses MBs. Containers are more efficient but share the kernel (slightly less isolated). VMs are fully isolated with their own kernel. In practice: containers for most microservices, VMs for stronger isolation requirements.

Q
Scenario: Production deploy fails and the site is down. What do you do first?
▼

Step 1: Restore service (rollback first, investigate second). Rollback to last working version immediately — every minute of downtime has business cost. Then: check service status (systemctl status), check logs (journalctl -u appname -n 50), check disk/memory, check if config changed. Once stable, do a postmortem on what went wrong and fix forward. Key principle: never investigate on a live broken system — restore first.

Q
What is Infrastructure as Code and why does it matter?
▼

IaC means defining infrastructure (servers, networks, databases) in code files (Terraform, CloudFormation, Ansible) rather than clicking through a GUI. Benefits: version-controlled (git history of every infrastructure change), reproducible (spin up identical environments), reviewable (code review for infrastructure changes), automated (no manual steps = fewer errors). It makes infrastructure auditable, testable, and recoverable — if a region fails, you can recreate it from code.

Q
How do you securely handle secrets (passwords, API keys) in shell scripts?
▼

Never hardcode secrets in scripts. Options in order of preference: (1) Environment variables from a secrets manager (AWS Secrets Manager, HashiCorp Vault). (2) Files with strict permissions (600) that are NOT in version control — reference them with source .env. (3) CI/CD secret variables (GitHub Secrets, GitLab CI variables) — these are injected at runtime, never stored in code. Always add .env to .gitignore. Audit git history if secrets were ever committed — they need to be rotated.
